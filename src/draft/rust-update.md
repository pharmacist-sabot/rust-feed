## ขอบเขตของหมวดนี้

บทความในหมวด Rust Update ครอบคลุมเนื้อหาประเภท:

- **Rust Release Notes** — สรุปฟีเจอร์ใหม่ใน Rust เวอร์ชันต่างๆ
- **RFC & Proposal Tracking** — ติดตาม RFC สำคัญที่กำลังจะเปลี่ยนแปลงภาษา
- **Project Goals & Roadmap** — ทิศทางการพัฒนาของ Rust Project
- **Ecosystem News** — ข่าวสารจาก Rust Foundation, Working Groups, Teams
- **Policy & Governance Changes** — การเปลี่ยนแปลงเชิงนโยบายและการบริหารโปรเจกต์
- **Edition Updates** — การเปลี่ยนแปลงระดับ Edition (เช่น Rust 2024, 2027)

## โครงร่างลำดับเนื้อหา

### 1. หัวข้อบทความ (Title)

```
# {ชื่อเรื่องภาษาไทยหรือผสม} — {Subtitle ที่สื่อถึงผลกระทบหรือความสำคัญ}
```

**แนวทาง:**
- ต้องสื่อให้ผู้อ่านเข้าใจทันทีว่า "สิ่งนี้คืออะไร" และ "ทำไมต้องสนใจ"
- ควรมี Keyword หลักของหัวข้อข่าว
- หลีกเลี่ยงชื่อที่คลุมเครือเกินไป

**ตัวอย่าง:**
- ดี: `Rust Project Goals 2026 — วิวัฒนาการครั้งสำคัญของกระบวนการพัฒนาที่ Rustacean ควรรู้`
- ดี: `จาก Feature สู่ Foundation: Rust มุ่งสู่การพัฒนาระดับโครงสร้างข้ามปี`
- ไม่ดี: `อัปเดต Rust` (คลุมเครือเกินไป)
- ไม่ดี: `Rust News ประจำสัปดาห์` (ไม่เจาะจงพอ)

### 2. บทนำ (Introduction) — 1-2 ย่อหน้า

**วัตถุประสงค์:** ให้บริบทว่าข่าว/อัปเดตนี้คืออะไร เกิดขึ้นเมื่อไหร่ ใครเป็นผู้ประกาศ และทำไมถึงสำคัญ

**ควรมี:**
- วันที่หรือช่วงเวลาของเหตุการณ์
- ผู้ที่เกี่ยวข้อง (เช่น ชื่อ Team, ชื่อบุคคลสำคัญ)
- แหล่งที่มาของข่าว (Blog post, RFC, GitHub Issue)
- ย่อหน้าสรุป "ทำไมต้องสนใจ" (Why it matters)

**ตัวอย่าง Blockquote เปิดเรื่อง:**
```
> **เมื่อ Rust โตพ้นช่วง "เพิ่มฟีเจอร์"**
>
> การเปลี่ยนแปลงครั้งนี้สะท้อนให้เห็นว่า Rust ไม่ได้แข่งกันที่ Syntax Sugar
> อีกต่อไป แต่กำลังเข้าสู่เฟสของการ **บูรณาการระดับ System**
```

### 3. เนื้อหาหลัก (Main Content) — 3-6 หัวข้อย่อย

แต่ละหัวข้อย่อยควรใช้ `##` heading และคั่นด้วย `---`

**ลำดับเนื้อหาที่แนะนำ:**

#### 3.1 ภาพรวมของการเปลี่ยนแปลง (What Changed)
- อธิบายว่าอะไรเปลี่ยน จากอะไรเป็นอะไร
- ใช้ Blockquote เน้นจุดสำคัญ
- ใส่ตารางเปรียบเทียบ Before/After ถ้าเหมาะสม

#### 3.2 รายละเอียดทางเทคนิค (Technical Details) — 1-3 หัวข้อ
- ลงลึกในแต่ละประเด็นสำคัญ
- แยกเป็นหัวข้อย่อยตามจำนวนประเด็น
- ใช้ Code snippet ประกอบเมื่อจำเป็น (แต่ไม่ต้องยาวเกินไป เน้นสรุปแนวคิด)
- ใช้ตาราง/bullet points เพื่อความชัดเจน

#### 3.3 ผลกระทบต่อ Developer (Impact)
- สิ่งนี้จะส่งผลอย่างไรต่อการเขียน Rust ในชีวิตประจำวัน
- ใครได้ประโยชน์ ใครต้องปรับตัว
- มี Breaking changes หรือไม่

#### 3.4 Timeline / กำหนดการ (ถ้ามี)
- ใช้ตารางแสดง Timeline ให้ชัดเจน
- ระบุว่า "เราอยู่ตรงไหน" ใน Timeline ปัจจุบัน

**ตัวอย่างตาราง Timeline:**
```
> | เดือน | กิจกรรม |
> | --- | --- |
> | **มกราคม** | รวบรวม proposals |
> | **กุมภาพันธ์** | รับ feedback ← **เราอยู่ตรงนี้** |
> | **มีนาคม** | เปิด RFC |
> | **เมษายน** | ประกาศอย่างเป็นทางการ |
```

#### 3.5 วิธีมีส่วนร่วม (How to Get Involved) — ถ้าเปิดให้ contribute
- ลิงก์ไปยัง RFC, Discussion, หรือช่องทางการมีส่วนร่วม
- อธิบายกระบวนการอย่างชัดเจน

### 4. บทสรุป (Conclusion) — 1-2 ย่อหน้า

**ควรมี:**
- สรุปประเด็นหลักทั้งหมดใน 2-3 ประโยค
- มุมมอง/ความคิดเห็นของผู้เขียนว่าสิ่งนี้สำคัญอย่างไรต่อ Rust community
- กระตุ้นให้ผู้อ่านติดตามหรือมีส่วนร่วมต่อ (Call to action)
